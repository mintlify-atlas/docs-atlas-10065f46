## What is the OpenAPI Specification?

The OpenAPI Specification (OAS) is a vendor-neutral, language-agnostic standard for describing HTTP APIs. A compliant OpenAPI document is a JSON or YAML file that describes an API's endpoints, request/response shapes, authentication schemes, and metadata. Tools can consume this document to generate client SDKs, server stubs, interactive documentation, and test suites without inspecting source code.

Swagger Core is the reference Java implementation of the OpenAPI Specification. It provides the annotations, model classes, scanning engine, and serialization layer that turn annotated Java classes into a valid OpenAPI document.

## OpenAPI 3.0 vs 3.1

Swagger Core supports both OpenAPI 3.0.x and 3.1.x. The two versions share the same overall structure but differ in several areas.

<Tabs>
  <Tab title="OpenAPI 3.0">
    - Schema objects follow a strict subset of JSON Schema Draft 07.
    - Nullable fields use `nullable: true` (an OAS extension, not part of JSON Schema).
    - `exclusiveMinimum` and `exclusiveMaximum` are booleans that modify the adjacent `minimum`/`maximum` values.
    - No native webhook support; webhooks must be modelled as regular paths with documentation-only conventions.
    - Schema `type` is a single string (`"string"`, `"integer"`, etc.).
  </Tab>
  <Tab title="OpenAPI 3.1">
    - Schema objects are fully aligned with JSON Schema Draft 2020-12.
    - Nullable types are expressed using a `type` array: `["string", "null"]`.
    - `exclusiveMinimum` and `exclusiveMaximum` are now numeric values, not booleans.
    - First-class **webhooks** support: the top-level `webhooks` field holds `PathItem` objects that describe incoming requests your API accepts from subscribers.
    - `$schema`, `$id`, `$anchor`, `if`/`then`/`else`, `prefixItems`, `unevaluatedProperties`, and other JSON Schema keywords are valid in schemas.
    - `@JsonIgnore` annotations suppress 3.0-only fields (such as `nullable`) during 3.1 serialization — see the `SortedSchemaMixin31` inner class in `GenericOpenApiContext`.
  </Tab>
</Tabs>

### Choosing a version in Swagger Core

Set `openAPI31 = true` on `SwaggerConfiguration` to emit a 3.1 document. The `convertToOpenAPI31` flag performs a post-processing pass through `OpenAPI31SpecFilter` to upgrade a 3.0 document built from annotations to 3.1.

```java
SwaggerConfiguration config = new SwaggerConfiguration()
    .openAPI31(true)          // emit openapi: "3.1.0"
    .resourcePackages(Set.of("com.example.api"));
```

To keep 3.0 output (the default), omit `openAPI31` or set it to `false`.

## How Swagger Core generates the spec

The generation pipeline has three stages:

```
┌─────────────────────────────────────────────────────────────┐
│  1. SCAN                                                     │
│  OpenApiScanner discovers resource classes via               │
│  resourcePackages / resourceClasses configuration            │
└────────────────────────┬────────────────────────────────────┘
                         │  Set<Class<?>>
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  2. READ                                                     │
│  Reader (JAX-RS) or custom OpenApiReader walks each class,  │
│  extracts @Path, @GET/@POST/@PUT…, @Operation, @Parameter,  │
│  @ApiResponse, @Schema etc. and builds Operation / PathItem │
│  objects. ModelConverters resolves Java types → Schema.      │
└────────────────────────┬────────────────────────────────────┘
                         │  OpenAPI model object
                         ▼
┌─────────────────────────────────────────────────────────────┐
│  3. SERIALIZE                                                │
│  GenericOpenApiContext uses Jackson ObjectMappers            │
│  (Json/Json31 for JSON, Yaml/Yaml31 for YAML) to write the  │
│  OpenAPI model to the wire format requested by the client.   │
└─────────────────────────────────────────────────────────────┘
```

### Stage 1 — Scanning

`GenericOpenApiContext.read()` calls `openApiScanner.classes()` and `openApiScanner.resources()` to discover the classes to process. The default scanner (`GenericOpenApiScanner`) uses [ClassGraph](https://github.com/classgraph/classgraph) to find all classes in the configured `resourcePackages`, or falls back to the explicit `resourceClasses` set.

### Stage 2 — Reading

The `Reader` class (`io.swagger.v3.jaxrs2.Reader`) implements `OpenApiReader`. For each discovered class it:

1. Checks for `@Hidden` on the class — if present, skips the entire class.
2. Reads `@OpenAPIDefinition` (if present) to populate top-level `info`, `servers`, `tags`, and `security`.
3. Iterates over all public methods, sorted deterministically.
4. Resolves the effective HTTP path by combining the class-level `@Path` with the method-level `@Path`.
5. Determines the HTTP verb from JAX-RS annotations (`@GET`, `@POST`, `@PUT`, `@DELETE`, `@PATCH`, `@HEAD`, `@OPTIONS`).
6. Merges `@Operation` annotation data (summary, description, operationId, tags, security, servers) with what was inferred from JAX-RS annotations.
7. Resolves method parameters and return types through `ModelConverters` to produce `Schema` objects stored in `components/schemas`.

### Stage 3 — Serialization

`GenericOpenApiContext` holds two pre-configured Jackson `ObjectMapper` instances — one for JSON (`Json`/`Json31`) and one for YAML (`Yaml`/`Yaml31`). The correct pair is selected during `init()` based on `isOpenAPI31()`. When `sortOutput` is enabled the mappers are additionally configured with `ORDER_MAP_ENTRIES_BY_KEYS` and `SORT_PROPERTIES_ALPHABETICALLY`.

## Spec output structure

A generated OpenAPI document contains these top-level sections:

| Section | Description |
|---|---|
| `openapi` | Version string — `"3.0.1"` or `"3.1.0"` |
| `info` | Title, description, version, contact, license |
| `servers` | Base URLs of the API |
| `paths` | Map of URL patterns to `PathItem` objects, each containing one `Operation` per HTTP verb |
| `components` | Reusable `schemas`, `securitySchemes`, `parameters`, `responses`, `requestBodies` |
| `tags` | Top-level tag descriptions |
| `webhooks` | (3.1 only) Map of webhook names to `PathItem` objects |

<Note>
The canonical field ordering used during sorted output is defined by `SortedOpenAPIMixin` (`openapi`, `info`, `externalDocs`, `servers`, `security`, `tags`, `paths`, `components`) and `SortedOpenAPIMixin31` (same plus `webhooks`) in `GenericOpenApiContext`.
</Note>

### Example: minimal generated JSON

```json
{
  "openapi": "3.0.1",
  "info": {
    "title": "Pet Store API",
    "version": "1.0.0"
  },
  "paths": {
    "/pets/{id}": {
      "get": {
        "operationId": "getPetById",
        "parameters": [
          {
            "name": "id",
            "in": "path",
            "required": true,
            "schema": { "type": "integer", "format": "int64" }
          }
        ],
        "responses": {
          "default": {
            "description": "default response",
            "content": {
              "application/json": {
                "schema": { "$ref": "#/components/schemas/Pet" }
              }
            }
          }
        }
      }
    }
  },
  "components": {
    "schemas": {
      "Pet": {
        "type": "object",
        "properties": {
          "id":   { "type": "integer", "format": "int64" },
          "name": { "type": "string" }
        }
      }
    }
  }
}
```

## Model conversion

Java types encountered as method return types, parameter types, or `@Schema`-annotated fields are converted to OpenAPI `Schema` objects by the `ModelConverters` singleton. `ModelConverters` maintains an ordered chain of `ModelConverter` instances. Each converter's `resolve(AnnotatedType, ModelConverterContext, Iterator<ModelConverter>)` method either handles the type or delegates to `chain.next()`.

The default converter is `ModelResolver`, which uses Jackson's introspection APIs to walk class properties, honour `@JsonProperty`, `@JsonIgnore`, and bean validation annotations, and produce the corresponding schema. Custom converters can be registered via `SwaggerConfiguration.modelConverterClasses`.
